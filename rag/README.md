# Retrieval-Augmented Generation (RAG) 아키텍처

## 1. 개요

RAG(Retrieval-Augmented Generation)는 산출물 생성 품질을 높이기 위해 행정기관 정보시스템 지침, DB 표준화 매뉴얼, UI/UX 가이드라인 등의 기준, 가이드 문서들을 벡터화하여 Qdrant Vector DB에 저장합니다.

산출물 생성 시 각 Agent는 목적에 맞는 검색 Query를 생성하고, 관련 문서를 검색하여 LLM Prompt Context에 주입한다. 이를 통해 생성 결과의 **표준성**(Standardization), **근거성**(Traceability), **일관성**(Consistency)을 향상시킵니다.

## 3. 청킹 전략 설계

### 3-1. 공통 설계 원칙

1. **패턴 기반 1차 분할**: 문서 구조에 맞는 정규식 패턴으로 의미 단위(조/섹션/Chapter/부록/소절)를 1차 경계로 인식
2. **메타데이터 전파**: 1차 청크(primary chunk)에서 title/section을 추출한 뒤, 길이 초과로 분할된 모든 서브 청크에 동일 메타데이터 전파 (섹션 제목이 후속 청크에서 사라지는 문제 방지)
3. **페이지 경계 무관**: 전체 텍스트 합산 후 패턴 분할 → 조항/섹션이 페이지를 넘어가도 분리되지 않음. 각 청크의 시작 offset을 역추적해 `payload.page`에 기록
4. **슬라이딩 윈도우 2차 분할**: MAX_CHARS 초과 시에만 `split_oversized_chunk`로 추가 분할

### 3-2. 파일별 청킹 파라미터

| 파일 | 청킹 패턴 | 페이지 필터 | MAX_CHARS / OVERLAP |
|---|---|---|---|
| 행정기관_지침 | `r"제\d+조"` | 없음 (전체) | 1500 / 150 |
| 표준프레임워크 | `r"□\s*.{2,20}"` | `page < 47` (붙임 제외) | 1500 / 150 |
| 요구사항_가이드 | `r"\d+\s*\nCHAPTER"` | `page < 466` (붙임1,2 제외) | 2500 / 200 |
| 사업관리_가이드 | `r"부록\s*\d{2}"`<br/>(전체 인식 후 03,04,08 필터) | `page >= 83` | 1500 / 150 |
| DB_표준화_매뉴얼 | `r"\d+\.\d+(?:\.\d+)?\s+\S"` | `11 <= page < 174` | 1500 / 150 |
| uiux_guideline | breadcrumb 마커 기반 페이지 분류(기존 유지) | `page >= 43` | 900 / 120 |
* '공공데이터 공통표준'은 .xlsx 파일로서 시트별, 행별로 청킹 진행

## 4. Collection 구조 설계

초기에는 산출물 에이전트마다 참조 collection을 1:1로 매칭하는 분리 방안을 검토했으나, `applies_to` 등 메타데이터 필터링으로 에이전트별 구분이 가능하므로 **단일 collection(`ALPLED_reference`)으로 통합**하는 방향으로 최종 결정했습니다. 중복 적재 없이 모든 문서를 하나의 collection에 적재하고, 쿼리 시 메타데이터 필터로 에이전트별 검색 범위를 제한합니다.

### 메타데이터 필터 구조

에이전트별 구분은 각 청크의 `applies_to` 필드를 기준으로 필터링합니다. `chunk_type`은 같은 에이전트 내에서도 제약사항/검수기준 등 세부 용도를 구분하는 데 활용합니다.

| 문서 | applies_to (에이전트 태그) | chunk_type (세부 용도) |
|---|---|---|
| 행정기관_지침 | `requirements_definition` | `constraint_reference` |
| 표준프레임워크 | `requirements_definition`, `architecture_design` | `constraint_reference` |
| 요구사항_가이드 | `requirements_definition` | `acceptance_criteria_reference` |
| 사업관리_가이드 부록03/04 | `requirements_definition` | `acceptance_criteria_reference` |
| 사업관리_가이드 부록08 | `requirements_definition` | `constraint_reference` |
| DB_표준화_매뉴얼 | `erd`, `database_design` | `db_standard_reference` |
| 공공데이터_공통표준 | `erd`, `database_design` | 시트별 `term_standard` / `word_standard` / `domain_standard` |
| UIUX_가이드라인 | `interface_design` | `design_principle`, `layout_rule`, `icon_rule` 등 |

## 6. 청킹 파라미터(MAX_CHARS) 평가

청킹 파라미터(MAX_CHARS) 수치 선정을 위해 2단계 평가를 설계했습니다.

### 6-1. 평가 방법론

| Layer | 방법 | 목적 |
|---|---|---|
| Layer1 | 1차 청크 길이 분포 분석 | 현재 MAX_CHARS 기준 추가 분할 비율,<br/>파일별 분포 특성 파악 |
| Layer2 | in-memory retrieval 비교 (bge-m3 + cosine similarity) | 대표 파일에서 MAX_CHARS 후보값별 recall@k / MRR 측정 |

### 6-2. Layer1 결과

| 파일 | 1차 청크 수 | 길이 분포 | 현재 MAX_CHARS 분할률 | 결론 |
|---|---|---|---|---|
| 행정기관_지침 | 108 | p95=450, max=744 | 0% | 자연단위 항상 보존 → 1500 유지 |
| 표준프레임워크 | 32 | 89~5,536, p95=5,113 | 46.9% | 분기점 → Layer2 실험 대상 선정 |
| 요구사항_가이드 | 18 | 4,131~187,648 | 100% | 자연단위 보존 불가능 (항상 분할) |
| 사업관리_가이드 | 3 (부록03/04/08) | 2,281~4,771 | 100% | 요구사항_가이드와 동일한 한계, N=3 |
| DB_표준화_매뉴얼 | 59 | 84~10,918, p95=8,125 | 33.9% | 표준프레임워크와 구조적으로 유사 |
| uiux_guideline | 515 (페이지 단위) | p95=678, max=935 | 0.2% | 자연단위 거의 항상 보존 → 900 유지 |

### 6-3. Layer2 결과: 표준프레임워크 MAX_CHARS 비교

**비교 대상 문서** : 표준프레임워크

선정 이유 : 길이 분포가 넓게 퍼져 있고 MAX_CHARS 초기 설정값(1500)이 분포의 중간 지점에 있어서, 후보값(900~3000)으로 바꿨을 때 chunk 구성이 실제로 달라지고 retrieval 결과가 달라질 가능성이 높습니다.

평가 지표 산출을 위한 **테스트 쿼리셋 설계 원칙**: Ground Truth를 섹션 제목이 아닌 **본문 content 기반**으로 설계했습니다. 제목 기반 GT는 쿼리와 거의 동일한 문구라 ceiling effect가 발생할 수 있어 실험 변별력이 저하됩니다.

**테스트 쿼리셋**:
표준프레임워크 문서의 1차 청크들 중 길이 분포 전반에 걸쳐 청크를 선정한 후 쿼리와 GT 문구를 작성했습니다.

| # | 쿼리 | 대상 청크(길이) | GT 문구 |
| --- | --- | --- | --- |
| 1 | 표준프레임워크의 라이선스 정책은? | 18 | 아파치(Apache) 라이선스 2.0 및 MIT 라이선스 |
| 2 | 적용 시 소스코드 수정 권고사항은? | 24 | 수정 없이 사용하여야 함 |
| 3 | 공통컴포넌트란? | 19 | 재사용이 가능하게 패키지로 제공하는 독립된 모듈 |
| 4 | 표준프레임워크 사용 법적 근거는? | 20 | 전자정부법 법률 제20654호 |
| 5 | 적용 가능 시스템 조건은? | 22 | 자바 기반의 웹 응용 시스템 |
| 6 | 적용 여부 확인 방법은? | 27 | 표준프레임워크 실행환경의 정상적인 설치 여부 점검 |
| 7 | 공통컴포넌트 재사용 가능 조건은? | 28 | 표준프레임워크를 기반으로 개발된/ 될 시스템 |
| 8 | 표준프레임워크 정의와 도입 효과는? | 03 | 필요한 기본 기능들을 표 준화 |
| 9 | 2.7 버전 실행환경 오픈소스 종류는? | 05 | (2.7기준) |
| 10 | 4.0 버전 실행환경 오픈소스 도구는? | 13 | (4.0기준) |

<br/>

**GT 유일성 검증**: 10개 GT 문구가 1차 청크들 중 의도한 1개에만 등장하는지 전수 확인했습니다.

<br/>

**실험 결과** (OVERLAP=150 고정, MAX_CHARS 단일 변수):

| MAX_CHARS | recall@1 | recall@3/5 | MRR |
|---|---|---|---|
| 900 | 0.80 | 0.90 | 0.838 |
| 1200 | 0.80 | 0.90 | 0.838 |
| **1500** | **0.90** | **0.90** | **0.905** |
| 1800 | 0.90 | 0.90 | 0.906 |
| 3000 | 0.80 | 0.90 | 0.840 |

<br/>

**세부 분석**:
- 900/1200: 쿼리 #4 (대상 청크 1,623자)가 3위로 밀림 → 윈도우가 작아 핵심 내용이 분할됨
- 3000: 쿼리 #10 (대상 청크 4,115자)이 3위로 밀림 → 윈도우가 너무 커서 헤더 신호 희석
- **1500/1800이 두 trade-off를 모두 회피하는 sweet spot**

<br/>

**결론**: 1500과 1800이 동률에 가깝게 우수하여 초기 설정값 1500을 유지하는 근거가 되었습니다.

## 7. 임베딩 모델 선택: `BAAI/bge-m3`

RAG 파이프라인에서 임베딩 모델은 청크 텍스트를 벡터로 변환하는 핵심 컴포넌트입니다. 단순히 성능 리더보드 상위 모델을 고르지 않고, **우리 코퍼스(한영 혼용 공공기술문서)와 파이프라인 요구사항에 맞는 모델**을 선택 기준과 실험으로 결정하고자 했습니다. 그 결과 최종적으로 선택한 모델은 `BAAI/bge-m3`입니다.

### 7-1. 선택 근거

#### 1. 한국어·영어 혼용 텍스트 처리

RAG 소스 문서들은 한국어 본문 안에 영어 기술 용어·제품명·라이선스명 등의 영문 토큰이 높은 밀도로 섞여 있습니다. (ex: "Spring Framework 5.3.20 √ Apache 2.0", "JavaEE(J2EE) 5") **bge-m3**는 100개 이상 언어로 학습된 multilingual 모델로서, 한국어 전용 모델과 달리 이런 한영 혼용 텍스트에서 영문 토큰의 의미도 같은 임베딩 공간 안에서 처리합니다.

#### 2. MAX_CHARS 범위와의 정합성

'6.청킹 파라미터 평가(Layer1/Layer2)'에서 확정한 MAX_CHARS는 900~2,500자입니다. 512 토큰 한도 모델(e5-large-instruct 등)은 청크 일부를 임베딩 시점에 잘라낼 수밖에 없습니다. **bge-m3**의 8,192 토큰 한도는 현재 청크 길이 범위를 충분히 수용합니다.

#### 3. 로컬 자체 호스팅

MIT 라이선스, `sentence-transformers` 기반 로컬 실행으로 외부 API에 텍스트가 전송되지 않습니다. 행정기관 지침·공공 SW 사업 가이드 등 공공 데이터 성격의 문서를 다루는 본 프로젝트에서, OpenAI 임베딩 API 같은 외부 전송 방식보다 적합합니다.

### 7-2. 대안 모델 비교

| 공개 시점 | 2024.02 | — | 2025.06 | — |

### 대안 미채택 이유

**`multilingual-e5-large-instruct`**: 최대 512 토큰 한도라서 현재 MAX_CHARS(900~2,500자) 범위의 청크를 임베딩 시점에 잘라낼 위험이 있습니다. 또한 모든 입력에 `"query:"` / `"passage:"` prefix를 붙여야 해 ingest·쿼리 파이프라인에 추가 처리 단계가 필요합니다.

**`Qwen3-Embedding-0.6B`**: 32,768 토큰으로 bge-m3보다 길지만, 현재 MAX_CHARS(최대 2,500자)는 8,192 토큰 안에도 충분히 들어가 이 여유가 실질적 이점으로 이어지지는 않습니다. 또한 최신 모델일수록 라이브러리 호환성 리스크가 있다는 우려(2025년 6월 공개)가 있고, 실제로 동일 환경에서 에러가 발생해 실행 자체가 불가능했습니다.

**`dragonkue/BGE-m3-ko`**: bge-m3를 한국어 데이터로 추가 학습한 모델로, 아키텍처·최대 토큰·차원이 원본과 동일합니다. 일반 한국어 검색 벤치마크에서는 bge-m3와 거의 동등하거나 근소하게 앞선다고 알려져 있으나, 본 코퍼스에서는 반대 결과가 나타났습니다.(→ 아래 실험 결과 참고)

### 7-3. 실험 결과

동일한 10개 테스트 쿼리셋, MAX_CHARS=1500(확정값) 고정 조건에서 세 모델을 in-memory(Qdrant 없이 cosine similarity)로 비교했습니다.

| Model | recall@1 | recall@3 | recall@5 | MRR |
|---|---|---|---|---|
| **`BAAI/bge-m3`** | **0.90** | **0.90** | **0.90** | **0.905** |
| `multilingual-e5-large-instruct` | 0.60 | 0.70 | 0.90 | 0.714 |
| `dragonkue/BGE-m3-ko` | 0.70 | 0.90 | 0.90 | 0.800 |

> `Qwen3-Embedding-0.6B`는 라이브러리 호환성 문제로 실행 불가, 결과 없음.

**해석 — e5**: 실제 순위에서 e5는 `#4(법적 근거)`, `#7(재사용 조건)`, `#8(정의/효과)` 쿼리가 2~4위로 밀려 recall@1이 낮아졌습니다. e5는 bge-m3 대비 컨텍스트 길이·instruction 요구사항·학습 방식이 모두 다르므로, 이 결과는 "현재 요구사항을 전체적으로 만족하는 대안이 있는가"에 대한 종합 비교로 해석하는 것이 좋습니다.

**해석 — dragonkue**: 아키텍처·컨텍스트 길이·차원이 동일하고 "한국어 추가 학습 여부"만 다른 조건이기 때문에, 7-1. 선택 근거 1번 **(한국어·영어 혼용 텍스트 처리)의 직접 검증**이 됩니다. 결과는 쿼리 유형별로 다르게 나타났습니다.

| 쿼리 | bge-m3 순위 | dragonkue 순위 | 해석 |
|---|---|---|---|
| #4 법적 근거 (전자정부법 조문) | **1위** | 6위 | 법률 조문 특유의 formal한 표현에서 원본 multilingual 임베딩이 더 적합 |
| #9 버전 식별자 `(2.7기준)` | 22위 | **3위** | lexical 패턴에는 한국어 추가 학습이 도움 |

그러나 전체적으로는 bge-m3가 더 안정적이며, 한국어 추가 학습이 본 코퍼스의 모든 텍스트 유형에 균일하게 도움이 되지는 않음을 확인했습니다.
